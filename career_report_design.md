## 生涯报告生成方案（草案）

### 目标
- 基于“学生画像 + 目标岗位 + 人岗匹配结果”，生成一份可落地的生涯报告
- 报告必须包含：差距分析、学习路径、实践安排（项目/实习）、分阶段计划（短期/中期）、评估周期与指标、动态调整规则
- 输出结构稳定、字段可控，便于前端渲染与后续迭代

### 适用范围与假设
- 岗位池固定为 10 个：阶段一先推荐 Top 3，用户从 Top 3 或“其他 7 个”中选择最终目标岗位
- 生涯报告在“用户已选择目标岗位”后生成（第二阶段）
- 后端已有：学生画像（StudentProfile）、岗位画像（job_profiles.json）、匹配评分（calculate_match）

### 输入与输出
#### 输入（后端生成报告所需上下文）
- student_profile：学生画像（阶段一产物，前端回传或后端缓存）
- target_job：目标岗位画像（从岗位库读取）
- match_detail：匹配详情（match_score + 维度拆解 + 规则/语义原因）

建议将上述三者整理成一个 report_context（用于 LLM 或模板引擎）：
```json
{
  "student_profile": { "...": "..." },
  "target_job": { "...": "..." },
  "match_detail": { "...": "..." },
  "generated_at": "2026-03-13T10:00:00Z",
  "report_version": "v1"
}
```

### 报告内容设计（核心原则）
- 以“可执行”为第一原则：每一项建议都必须有可验收产物（deliverable/acceptance）
- 以“个性化”为第二原则：引用学生画像证据（evidence）解释为什么推荐该路线
- 以“动态调整”为第三原则：指标可量化、周期固定、规则可触发

### 分阶段计划模板（示例逻辑）
#### 短期（2-4周）：补齐关键短板 + 建立作品集骨架
- 目标：补齐目标岗位最关键 1-2 个硬技能短板；完成一个可展示的小项目或现有项目重构
- 学习路径：聚焦“岗位硬要求”主题，输出：可运行的 demo、笔记、技术文章
- 实践安排：
  - 项目：围绕岗位场景做一个“可演示”的项目模块（如高并发/检索/推荐/权限）
  - 实习：开始准备简历迭代 + 基础面试题库；每周至少一次模拟面试
- 周度节奏：每周固定 1 次复盘（30-60分钟），产出：周报 + 下周计划

#### 中期（1-3个月）：能力结构化提升 + 实习/投递闭环
- 目标：形成“可复用的方法论”（系统设计、项目拆解、技术选型）；拿到至少 1 次真实面试反馈或实习机会
- 学习路径：从“会做”升级到“会解释 + 会权衡”，输出：设计文档、性能对比、复盘报告
- 实践安排：
  - 项目：完成一个“可写入简历”的完整项目/模块（含监控、压测、文档）
  - 实习：建立投递漏斗（投递-笔试-一面-二面-Offer），持续迭代简历与项目描述
- 里程碑：第 4/8/12 周分别产出一次“作品集更新 + 复盘”

### 评估周期与指标（建议）
#### 周期
- 建议默认：每周小复盘 + 每两周大复盘
- 每次复盘必须产出：指标数据、完成情况、问题列表、调整动作

#### 指标（可选组合）
1) 学习指标
- 主题掌握度：完成度（%）、测验正确率（%）、可复述程度（1-5）
- 资料输入产出比：学习小时数 vs 可交付产物数（笔记/代码/文章）

2) 实践指标
- 项目推进度：里程碑完成数/计划数；关键模块是否可演示
- 工程质量：单元测试覆盖（可选）、接口文档完善度、性能基线（如 QPS/延迟）

3) 求职指标（如进入投递/面试阶段）
- 投递数、笔试通过率、一面通过率、面试反馈主题分布
- 简历迭代次数、项目亮点量化指标数量（如“QPS 提升 30%”）

4) 匹配指标（与现有评分机制对齐）
- 7 维能力分：短期以“技能/证书/实习”提升为主，中期兼顾“素养/潜力”
- 目标岗位匹配分变化：报告可计算“本周期预期提升区间”（非强制）

### 动态调整机制（关键）
建议用“规则优先 + LLM 辅助”的策略：
- 规则优先：当指标触发阈值时，执行明确的调整动作（减少范围/拆解任务/换资源/增加实践）
- LLM 辅助：在规则无法给出方案时（例如动机不足、目标不清），用 LLM 输出多备选路径，并要求选择其一

调整动作的基本类型：
- 调整目标：短期目标过大则拆分；中期目标偏离岗位则回归关键能力
- 调整学习资源：更换为更适配的课程/书/文档
- 调整实践策略：从“做新项目”改为“重构已有项目并补齐文档/压测”
- 调整节奏：将周计划改为“2-3天一个小迭代”，降低拖延成本

### 生成策略（实现建议）
建议采用“结构化生成 + 校验 + 兜底模板”三段式：
1) 结构化上下文构建（后端代码）
- 从 student_profile/match_detail/target_job 提取最关键的信息（技能关键词、短板、维度分、岗位要求）
- 形成 report_context，控制长度与字段

2) 报告生成（LLM 或模板）
- 优先：LLM 生成结构化 JSON（严格限定字段）
- 备选：模板生成（完全不依赖 LLM），用于 LLM 失败兜底

3) 输出校验（后端）
- JSON schema/字段校验：缺字段补默认值，非法值回退模板
- 文本安全：不输出敏感信息；不输出不确定的断言（如“你一定能拿 Offer”）
