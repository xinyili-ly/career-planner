# `/api/v1/match/report` 接口详细文档

## 一、接口概述

**接口地址**：`POST /api/v1/match/report`  
**接口功能**：生成深度职业报告，包含人岗匹配分析（module_1）、职业路径规划（module_2）、行动计划（module_3）  
**特点**：支持分步生成，module_4（最终润色报告）需通过 `/api/v1/match/report/finalize` 单独获取

## 二、调用方式

支持两种方式（二选一）：

### 方式 1：直接传 `student_profile`
适用于前端已持有完整学生画像对象，无需缓存。

### 方式 2：传 `profile_cache_id`（推荐）
由 `/api/v1/parse/parse_manual` 接口返回的画像缓存 ID，可减少数据传输量，后端可直接从缓存读取。

**优先级**：若同时传入 `student_profile` 和 `profile_cache_id`，优先使用 `student_profile`。

## 三、请求参数

### 请求体（JSON）

```json
{
  "job_id": "JOB_001",
  "profile_cache_id": "3a74cc6b-3bc0-459f-a77f-bd597313167d"
}
```

> **说明**：行动计划（module_3）由 LLM 根据学生画像与岗位画像的差距自动生成，无需前端传入偏好设置。

### 参数说明

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `job_id` | string | **必填** | 目标岗位 ID（与 `/recommend/list` 返回的 `id` 一致） |
| `profile_cache_id` | string | **二选一** | 学生画像缓存 ID（推荐，由 parse/parse_manual 返回） |
| `student_profile` | object | **二选一** | 完整学生画像对象（见数据结构说明） |
| `precomputed_module_1` | object | 否 | 前端预计算的 module_1（跳过 LLM 调用，提升性能） |

## 五、核心字段说明

### 1. `summary`（顶层摘要）
最外层直接返回的摘要信息，前端无需进入 `career_report` 即可快速展示。

| 字段 | 类型 | 说明 |
|------|------|------|
| `student_name` | string | 学生姓名 |
| `target_job` | object | 目标岗位（job_id + title） |
| `match_score` | number | 总体匹配分（0-100） |
| `match_level` | string | 匹配等级（高度匹配、中度匹配、一般、不匹配） |
| `overall_fit_percent` | number | 同 `match_score` |
| `top_strengths` | array<string> | 核心优势标签（从 module_1.strengths 提取） |
| `key_gaps` | array<string> | 关键差距标签（从 module_1.gaps 提取） |

### 2. `job_info`（精简岗位信息）
仅包含前端展示必需字段，不包含冗长的 `description` 和 `keywords`。

### 3. `match_summary`（匹配摘要）
顶层返回的维度分数，比 `match_detail` 更精简。

### 4. `career_report`（报告主体）
包含三个核心模块（module_1~3），`module_4` 为 `null` 需单独调用 finalize 获取。

| `completion_rate` | number | 完成率（0-1） |
| `updated_at` | string | 更新时间（ISO 8601） |

## 六、性能优化提示

### 1. 利用 `precomputed_module_1` 跳过 LLM 调用
如果前端已通过其他途径（如本地缓存或 `match/detail` 接口）获得了 module_1 数据，可将其传入 `precomputed_module_1` 参数，后端将直接使用而无需再次调用 LLM，显著提升响应速度。

### 2. 优先使用 `profile_cache_id`
`profile_cache_id` 方式：
- 减少网络传输量（不传完整画像）
- 后端可直接从缓存读取，无需重复解析
- 保持画像一致性（同一缓存 ID 多次调用结果相同）

### 3. 接口幂等性
相同参数多次调用，返回结果一致（除了 `updated_at` 和 `plan_id` 可能变化）。

## 七、调用流程示例

### 1. 基础调用（使用 profile_cache_id）

```javascript
// 调用接口生成报告
const response = await fetch('http://192.168.31.92:8002/api/v1/match/report', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Accept': 'application/json'
  },
  body: JSON.stringify({
    job_id: 'JOB_001',
    profile_cache_id: '3a74cc6b-3bc0-459f-a77f-bd597313167d'
  })
});

if (!response.ok) {
  const error = await response.json();
  throw new Error(error.detail || '接口调用失败');
}

const reportData = await response.json();

// 使用报告数据
console.log('匹配分数:', reportData.summary.match_score);
console.log('优势:', reportData.summary.top_strengths);
console.log('差距:', reportData.summary.key_gaps);
```

### 2. 完整流程（先生成报告，再润色）

```javascript
async function generateFullReport(profileCacheId, jobId) {
  // 步骤1：生成报告（module_1~3）
  const reportResponse = await fetch('/api/v1/match/report', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      profile_cache_id: profileCacheId,
      job_id: jobId
    })
  });
  
  if (!reportResponse.ok) {
    throw new Error('报告生成失败');
  }
  
  const reportData = await reportResponse.json();
  
  // 立即展示 module_1~3
  displayMatchAnalysis(reportData.career_report.module_1);
  displayCareerPaths(reportData.career_report.module_2);
  displayActionPlan(reportData.career_report.module_3);
  displayTodoList(reportData.action_plan);
  
  // 步骤2：请求最终润色报告（module_4）
  const finalizeResponse = await fetch('/api/v1/match/report/finalize', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      profile_cache_id: profileCacheId,
      job_id: jobId,
      preview_data: reportData  // 传入 report 接口的完整返回
    })
  });
  
  if (finalizeResponse.ok) {
    const finalizeData = await finalizeResponse.json();
    // 将 module_4 的 polished_markdown 渲染到页面
    displayPolishedReport(finalizeData.module_4.polished_markdown);
  }
  
  return reportData;
}

// 调用
generateFullReport('3a74cc6b-...', 'JOB_001');
```

## 八、错误处理

### 常见错误码

| HTTP 状态码 | 错误信息 | 原因及解决方案 |
|-------------|----------|---------------|
| 400 | Bad Request | 请求参数格式错误，检查 JSON 结构 |
| 404 | 岗位不存在 | `job_id` 无效，确认岗位 ID 正确 |
| 422 | `preview_data` 不能为空 | `finalize` 接口未传入 `preview_data` |
| 500 | Internal Server Error | 服务器内部错误，查看后端日志 |
| 503 | Service Unavailable | 服务暂时不可用，稍后重试 |

### 错误响应格式

```json
{
  "detail": "岗位不存在"
}
```

### 错误处理示例

```javascript
async function safeCallReport(profileCacheId, jobId) {
  try {
    const response = await fetch('/api/v1/match/report', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        profile_cache_id: profileCacheId,
        job_id: jobId
      })
    });
    
    if (!response.ok) {
      const error = await response.json().catch(() => ({}));
      
      switch (response.status) {
        case 400:
          throw new Error(`请求参数错误: ${error.detail}`);
        case 404:
          throw new Error(`岗位不存在，请确认 job_id: ${jobId}`);
        case 500:
        case 503:
          throw new Error(`服务暂时不可用，请稍后重试`);
        default:
          throw new Error(`接口错误 (${response.status}): ${error.detail || '未知错误'}`);
      }
    }
    
    return await response.json();
    
  } catch (err) {
    console.error('报告生成失败:', err);
    // 可以选择：重试、上报到监控系统、展示友好提示
    throw err;
  }
}
```

## 十、模块状态字段说明

每个模块（module_1~4）都包含 `llm_status` 字段，用于标识 LLM 调用状态：

| 状态值 | 含义 | 前端处理建议 |
|--------|------|-------------|
| `"ok"` | 正常生成 | 直接使用返回数据 |
| `"skipped"` | 已跳过 | 数据来自缓存或预计算 |
| `"error"` | 生成失败 | 显示占位内容，可提示用户重试 |
| `"pending"` | 生成中 | 需等待或轮询 |

```javascript
// 检查模块状态
const module1Status = reportData.career_report.module_1?.llm_status;
if (module1Status === 'ok') {
  displayModule1(reportData.career_report.module_1);
} else if (module1Status === 'error') {
  showFallbackContent('module1');
}
```
